# The Observer & Sandbox Guardrails 🛡️

ErnOS V5 takes the concept of LLM-driven autonomy to its maximum logical extent, providing the agent with raw read/write access to local filespaces, active terminal command capability, and unbridled web access. 

Because OpenClaw previously suffered from severe command injection and Localhost Trust exploits, ErnOS V5 completely replaces the standard pipeline with a hybridized defense-in-depth model starring **The Observer**.

---

## 1. The Observer Pattern

LLMs are inherently susceptible to Prompt Injection, Sycophancy, and Hallucination. To combat this, V5 divides cognition into a primary **Actor** and a secondary **Observer**.

### How it Works
1. The **Actor** receives a user prompt, reasons about the context, and generates a JSON-formatted `.call()` to execute a tool (e.g., `fs_write` or `run_command`).
2. *Before* the tool ever reaches the execution engine, the payload is intercepted by the **Observer**.
3. The Observer is an isolated, secondary LLM call (or local rule-engine check) that holds strict, unmodifiable system directives. It evaluates the Actor's proposed action against a matrix of safety heuristics:
   - *Is the Actor attempting to modify `/etc/` or `~/.ssh/`?*
   - *Did an external web-fetch prompt the Actor to abruptly download and run an unknown binary?*
   - *Is the Actor hallucinating a tool that doesnt exist?*

If the Observer flags a critical breach, the tool call is rejected, an error is appended to the Actor's context window, and the Actor is forced to rethink its approach.

---

## 2. Docker Bind Mount Isolation

Even if the Observer fails (e.g., through an advanced jailbreak), the physical execution environment is severed from the host machine. 

ErnOS utilizes the `openclaw/sandbox` specification to drop *all* CLI, filesystem, and Node tool executions into a `non-main` Docker container.
- **Strict Network Segmentation**: Default runs use internal virtual bridges. Local LAN IP ranges (e.g., `192.168.x.x` or `10.x.x.x`) are actively blocked by the `fetchWithSsrFGuard` middleware, neutralizing SSRF vulnerabilities.
- **Vetted Bind Mounts**: The `validate-sandbox-security.ts` script aggressively parses any requested host mounts. Requests to mount the Docker socket (`/var/run/docker.sock`) or root filesystem pathways are explicitly dropped before container instantiation.

## 3. Cryptographic Anti-Gaslighting

When the agent autonomously generates a new script or image, it writes the file into the `workspace` directory. 

Historically, users could not determine whether an altered file was the work of the Agent, a fellow operator, or a third-party vulnerability.

V5 introduces the `ArtifactRegistry`. 
- Every file cleanly generated by a registered agent tool is dynamically hashed via `SHA-256`. 
- This hash is appended to an immutable, local JSON ledger.
- Using the `verify_artifact_authorship` tool, ErnOS can cryptographically prove whether it was the original author of any file found on disk, preventing gaslighting scenarios natively.
